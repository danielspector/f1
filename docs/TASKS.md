### T-03 · Define the database schema

> All data models are defined here as a single migration. Getting the schema right before writing any service code prevents painful migrations later.

**Depends on:** T-02

- [x] **T-03.1** · Define the `User` model
  ```prisma
  model User {
    id            String   @id @default(cuid())
    email         String   @unique
    passwordHash  String
    name          String?
    createdAt     DateTime @default(now())
    updatedAt     DateTime @updatedAt
    leagueMembers LeagueMember[]
    picks         Pick[]
    scores        PlayerScore[]
  }
  ```

- [x] **T-03.2** · Define the `League` and `LeagueMember` models
  ```prisma
  model League {
    id          String        @id @default(cuid())
    name        String
    inviteCode  String        @unique @default(cuid())
    seasonYear  Int
    status      LeagueStatus  @default(ACTIVE)
    createdById String
    createdAt   DateTime      @default(now())
    members     LeagueMember[]
    picks       Pick[]
    scores      PlayerScore[]
  }

  enum LeagueStatus { ACTIVE ARCHIVED }

  model LeagueMember {
    id        String     @id @default(cuid())
    leagueId  String
    userId    String
    role      MemberRole @default(MEMBER)
    joinedAt  DateTime   @default(now())
    league    League     @relation(fields: [leagueId], references: [id])
    user      User       @relation(fields: [userId], references: [id])
    @@unique([leagueId, userId])
  }

  enum MemberRole { MEMBER ADMIN }
  ```

- [x] **T-03.3** · Define the `Race` and `Seat` models
  ```prisma
  model Race {
    id           String       @id @default(cuid())
    seasonYear   Int
    round        Int
    name         String
    circuit      String
    fp1Deadline  DateTime     // UTC datetime of FP1 session start
    raceDatetime DateTime
    results      RaceResult[]
    picks        Pick[]
  }

  model Seat {
    id          String       @id @default(cuid())
    seasonYear  Int
    teamName    String
    driverName  String
    driverCode  String       // e.g. "VER", "HAM"
    results     RaceResult[]
    picks       Pick[]
    @@unique([seasonYear, driverCode])
  }
  ```

- [x] **T-03.4** · Define the `Pick` model
  ```prisma
  model Pick {
    id          String   @id @default(cuid())
    leagueId    String
    userId      String
    raceId      String
    seatId      String
    submittedAt DateTime @default(now())
    league      League   @relation(fields: [leagueId], references: [id])
    user        User     @relation(fields: [userId], references: [id])
    race        Race     @relation(fields: [raceId], references: [id])
    seat        Seat     @relation(fields: [seatId], references: [id])
    score       PlayerScore?
    @@unique([leagueId, userId, raceId]) // one pick per player per race per league
  }
  ```

- [x] **T-03.5** · Define the `RaceResult` and `PlayerScore` models
  ```prisma
  model RaceResult {
    id       String @id @default(cuid())
    raceId   String
    seatId   String
    position Int?   // null = DNF/DNS
    points   Int    @default(0)
    race     Race   @relation(fields: [raceId], references: [id])
    seat     Seat   @relation(fields: [seatId], references: [id])
    @@unique([raceId, seatId])
  }

  model PlayerScore {
    id           String  @id @default(cuid())
    leagueId     String
    userId       String
    raceId       String
    pickId       String  @unique
    pointsEarned Int     @default(0)
    league       League  @relation(fields: [leagueId], references: [id])
    user         User    @relation(fields: [userId], references: [id])
    pick         Pick    @relation(fields: [pickId], references: [id])
    @@unique([leagueId, userId, raceId])
  }
  ```

- [x] **T-03.6** · Run and verify the initial migration
  - Run `npx prisma migrate dev --name init`
  - Inspect the generated SQL in `prisma/migrations/` to confirm all tables and constraints look correct
  - Run `npx prisma studio` and verify all tables are visible and empty

### T-05 · Build the F1 data service

> All game logic depends on having accurate race schedules, driver rosters, and race results. Build and seed this before the game loop.

**Depends on:** T-03

- [x] **T-05.1** · Explore the Jolpica API
  - Base URL: `https://api.jolpi.ca/ergast/f1/`
  - Key endpoints to use:
    - `/{season}/races.json` — full race schedule with session times
    - `/{season}/drivers.json` — all drivers for a season
    - `/{season}/{round}/results.json` — race results for a specific round
  - Confirm session times are returned in UTC
  - Document the response shape for each endpoint in a comment block in `services/f1DataService.ts`

- [x] **T-05.2** · Create the `F1DataService` module
  - Create `services/f1DataService.ts`
  - Implement `fetchSeasonSchedule(year: number)` — returns array of races with FP1 datetime and race datetime
  - Implement `fetchDriverRoster(year: number)` — returns array of `{ driverCode, driverName, teamName }` for each seat
  - Implement `fetchRaceResults(year: number, round: number)` — returns array of `{ driverCode, position, points }`
  - Add a simple in-memory cache (or use `next/cache` `unstable_cache`) to avoid redundant API calls within the same request cycle
  - Handle API errors gracefully: log and rethrow with a typed `F1ApiError`

- [x] **T-05.3** · Create the database seed script
  - Create `prisma/seed.ts`
  - Call `F1DataService.fetchSeasonSchedule(currentYear)` and upsert all `Race` records
  - Call `F1DataService.fetchDriverRoster(currentYear)` and upsert all `Seat` records
  - The FP1 deadline stored on the `Race` record should be the exact UTC datetime of FP1 session start
  - Register seed script in `package.json`: `"prisma": { "seed": "ts-node prisma/seed.ts" }`
  - Run `npx prisma db seed` and verify records in Prisma Studio

- [x] **T-05.4** · Create an internal admin API to re-sync F1 data
  - Create `app/api/admin/sync-season/route.ts` (POST, protected by a secret header `x-admin-key`)
  - Calls `F1DataService` and upserts fresh data — useful for mid-season driver roster changes
  - This is not user-facing; it is called manually or via cron when needed

### T-06 · Build league management API

> Leagues are the top-level container for everything. API must be solid before building the UI.

**Depends on:** T-04, T-05

- [x] **T-06.1** · Create league API endpoint
  - Create `app/api/leagues/route.ts` (POST)
  - Authenticated endpoint — reject if no session
  - Body: `{ name: string, seasonYear: number }`
  - Generate a unique `inviteCode` (use `crypto.randomUUID()`)
  - Create `League` record and create a `LeagueMember` record for the creator with `role: ADMIN`
  - Return the created league including the `inviteCode`

- [x] **T-06.2** · Join league via invite code endpoint
  - Create `app/api/leagues/join/route.ts` (POST)
  - Body: `{ inviteCode: string }`
  - Validate the league exists and is `ACTIVE`
  - Check the user is not already a member (idempotent — return success if already joined)
  - Create a `LeagueMember` record with `role: MEMBER`
  - Return the league details so the UI can redirect

- [x] **T-06.3** · Get leagues for current user
  - Create `app/api/leagues/route.ts` (GET)
  - Return all leagues the current user is a member of, including their role and the league's current status
  - Include aggregate data per league: user's total points, current rank, and whether they have a pick for the upcoming race

- [x] **T-06.4** · Get single league details
  - Create `app/api/leagues/[id]/route.ts` (GET)
  - Return league info, all members, and the full leaderboard (each member's total points)
  - Verify the requesting user is a member; return 403 if not

- [x] **T-06.5** · Admin: update league settings
  - Create `app/api/leagues/[id]/route.ts` (PATCH)
  - Verify requesting user has `role: ADMIN` in this league; return 403 if not
  - Allowed operations: `{ name?: string }` to rename the league
  - Return updated league

- [x] **T-06.6** · Admin: manage members
  - Create `app/api/leagues/[id]/members/[userId]/route.ts`
  - DELETE — remove a member from the league (admin only; cannot remove yourself if last admin)
  - PATCH — promote/demote member role (`{ role: 'ADMIN' | 'MEMBER' }`) (admin only)
  - Always ensure at least one ADMIN remains in the league

- [x] **T-06.7** · Admin: archive league
  - Create `app/api/leagues/[id]/archive/route.ts` (POST)
  - Admin only
  - Set `League.status = ARCHIVED`
  - This is also called automatically by the end-of-season cron job (T-16)

- [x] **T-06.8** · Admin: renew league for new season
  - Create `app/api/leagues/[id]/renew/route.ts` (POST)
  - Admin only; source league must be `ARCHIVED`
  - Body: `{ seasonYear: number }`
  - Create a new `League` record with a new `inviteCode`, same `name`, new `seasonYear`
  - Copy all `LeagueMember` records (preserving roles) to the new league
  - Return the new league

### T-08 · Build the pick system API

> This is the heart of the game. Every validation rule from the PRD must be enforced server-side — never trust the client.

**Depends on:** T-05, T-06

- [x] **T-08.1** · Build the seat availability service
  - Create `services/pickService.ts`
  - Implement `getAvailableSeats(leagueId, userId, raceId)`:
    - Fetch all seats for the current season
    - Fetch all prior picks for this user in this league
    - If the user has used all seats → reset: all seats are available
    - Otherwise: return seats the user has not yet picked in this league
    - Always exclude the seat the user has already picked for this specific race (if they've already submitted)
  - Implement `hasUsedAllSeats(leagueId, userId, seasonYear)` → boolean
  - Unit test this function with edge cases: first pick, all-but-one used, full reset

- [x] **T-08.2** · Build the deadline enforcement utility
  - Create `lib/deadlines.ts`
  - Implement `isFP1Passed(race: Race): boolean` — compares `race.fp1Deadline` to `new Date()` in UTC
  - Implement `getUpcomingRace(races: Race[]): Race | null` — returns the next race whose FP1 has not yet passed
  - All deadline checks must use UTC to avoid timezone bugs

- [x] **T-08.3** · Create the pick submission API endpoint
  - Create `app/api/leagues/[id]/picks/route.ts` (POST)
  - Body: `{ raceId: string, seatId: string }`
  - Validations (return 400/403 with descriptive error on failure):
    1. User is a member of this league
    2. FP1 deadline has not passed for this race (`isFP1Passed` check)
    3. User has not already submitted a pick for this race in this league
    4. The seat is available for this user (using `getAvailableSeats`)
  - Create `Pick` record
  - Return the created pick

- [x] **T-08.4** · Create the get picks API endpoint
  - Create `app/api/leagues/[id]/picks/route.ts` (GET)
  - Query param: `?raceId=` (optional — returns all picks for league if omitted)
  - If `raceId` provided and FP1 has **not** passed: return only the current user's pick (hide others)
  - If `raceId` provided and FP1 **has** passed: return all members' picks for that race
  - Always return the current user's used-seat list so the UI can mark unavailable seats

## Phase 5 — Scoring & Leaderboard

---

### T-10 · Build the scoring service

> Scoring is triggered after each race. Results must flow from the F1 API through to every player's score in every league.

**Depends on:** T-08

- [x] **T-10.1** · Implement race result ingestion
  - In `services/scoringService.ts`, implement `ingestRaceResults(seasonYear, round)`:
    - Call `F1DataService.fetchRaceResults(year, round)`
    - Upsert `RaceResult` records for each seat (position + points)
    - Handle DNF/DNS: store `position: null, points: 0`
    - Only ingest the main race result — filter out sprint results

- [x] **T-10.2** · Implement points calculation
  - Implement `calculateScoresForRace(raceId)`:
    - Fetch all `Pick` records for this race (across all leagues)
    - For each pick, find the corresponding `RaceResult` via `seatId`
    - Create or update a `PlayerScore` record with `pointsEarned = raceResult.points`
    - If no pick exists for a member (missed deadline): ensure a `PlayerScore` of 0 is recorded
  - F1 standard points map: `{ 1: 25, 2: 18, 3: 15, 4: 12, 5: 10, 6: 8, 7: 6, 8: 4, 9: 2, 10: 1 }`

- [x] **T-10.3** · Create the race results ingestion API endpoint
  - Create `app/api/admin/ingest-results/route.ts` (POST, protected by `x-admin-key` header)
  - Body: `{ seasonYear: number, round: number }`
  - Calls `ingestRaceResults` then `calculateScoresForRace`
  - Returns a summary: how many results ingested, how many player scores updated
  - This endpoint is called by the post-race cron job (T-15)

- [x] **T-10.4** · Build the leaderboard data query
  - In `services/leagueService.ts`, implement `getLeaderboard(leagueId)`:
    - Return all members sorted by total `PlayerScore.pointsEarned` (descending)
    - Include per-member race history: array of `{ race, seatPicked, pointsEarned }`
    - This is the single source of truth for the Leaderboard and PlayerBreakdown components
